## 5. Architecture & Design Assessment
The stage-pipeline decomposition is good: single entry point, `RunContext`, stage modules, prompt builders, and update mapping are all separated cleanly. Contracts are generally enforced with zod plus semantic validators, but enforcement is inconsistent: `modulePlanSchema` is weaker than runtime expectations, `overviewGenerationResultSchema` is weaker than the spec, and `DocumentationStage` no longer matches the documented API. The biggest tech-design divergences are the missing real adapter, lack of module batching, missing SDK prerequisite checks in the environment stage, and no prior-vs-fresh relationship diff support for update mapping.

## 6. Test Quality Assessment
The tests are meaningful in the sense that they exercise filesystem writes, result assembly, validation passes, and update-side file timestamps rather than only asserting mocks. The weak spots are that every orchestration suite mocks `createAgentSDKAdapter`, so the production stub ships undetected; the empty-module test bypasses the real planner; schema-negative coverage exists for clustering but not for module generation, overview generation, or review patches; and the pure-function tests promised by the test plan are missing.

## 7. Incremental Update Path
`readPriorGenerationState` is solid for missing/malformed metadata and missing plan files in [prior-state.ts#L20](/Users/leemoore/code/code-steward/code-wiki-gen/src/orchestration/update/prior-state.ts#L20). `mapToAffectedModules` correctly handles direct file-to-module mapping, renames, deletions, module removal, and the ">50% affected" warning, but it is not robust for semantic corruption in the persisted plan, import-chain-based mapping, or removed relationship edges. The overview regeneration trigger is deterministic and correct for module removal in [affected-module-mapper.ts#L165](/Users/leemoore/code/code-steward/code-wiki-gen/src/orchestration/update/affected-module-mapper.ts#L165).

## 8. Recovery & Failure Handling
Most stage failures are mapped to structured results, partial outputs are intentionally preserved, and metadata rollback is thoughtfully handled. The main failure-handling problems are that SDK init is misattributed to the wrong stage, `quality-review` progress is emitted late, and metadata snapshot reads can still throw past the `EngineResult` boundary. Error messages are generally actionable, and the no-retry behavior matches the design's "retry deferred" decision.
