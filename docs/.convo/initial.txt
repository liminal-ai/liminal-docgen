  - Step
  - Actor/component
  - Action
  - Output / side effect

  On determinism: we should want more of it, but not maximize it blindly.

  What determinism buys us:

  - repeated runs on the same repo produce the same module boundaries and page names
  - updates become safer because module identity is more stable
  - docs stop “thrashing” between runs
  - humans and agents can trust links and references more

  What we sacrifice if we over-index on it:

  - adaptiveness
      - the planner becomes less willing to discover a better grouping
  - quality of clustering
      - if we freeze module shapes too aggressively, we can lock in mediocre boundaries
  - expressiveness
      - summaries and packets can become flatter and more template-like
  - sensitivity to real structure
      - two codebases with genuinely different shapes may get forced through the same conservative mold

  So I would not chase “perfect determinism everywhere.”
  I would chase:

  - deterministic naming
  - deterministic packet eligibility
  - deterministic downgrade behavior
  - mostly stable module boundaries
  - while still allowing some flexibility when the repo genuinely supports a better grouping

  The right balance is:

  1. make the system deterministic on the low-level mechanics

  - file naming
  - packet mode
  - section order
  - validation rules
  - repair/downgrade behavior

  2. make planning bounded, not frozen

  - allow module planning to choose among good shapes
  - but constrain it enough that it doesn’t wildly re-split the repo every run

  3. stabilize the high-value identities

  - if a module like Convex Auth & RLS exists in one run, it shouldn’t become Convex Auth & Security in the next unless the repo structure truly
    changed

  - module naming normalization
  - stable grouping heuristics for obvious subsystems

  - maybe canonical merge/split rules for borderline modules

  What I would not do:

  - hard-freeze the exact full module plan too early
  - make all summaries/template text deterministic at the cost of readability
  - eliminate inference judgment completely from clustering

  - push determinism into identity and boundaries
  - not all the way into content quality and clustering creativity

  - much more stable docs
  - without turning the whole thing into a rigid, low-quality summarizer
