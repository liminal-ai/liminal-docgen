## Context

This is the core architectural change in the epic: replacing one-shot structured-output generation with a multi-turn agent that reads code, makes decisions, and writes documentation sections iteratively.

Today, `generateModulePage()` in `src/orchestration/stages/module-generation.ts` follows a rigid path. It calls `selectModuleDocumentationPacket()` to predict what sections the model should produce (574 lines of scoring heuristics that decide full-packet vs summary-only mode, whether a sequence diagram is warranted, what structure diagram kind to use). It calls `buildModuleDocumentationFacts()` to assemble deterministic context (entity candidates from exports and relationships, flow candidates from cross-module edges, relationship summaries). It calls `buildModuleDocPrompt()` to construct the prompt. It calls `provider.infer()` once with a Zod schema, and the model must return a complete JSON payload in a single response. When the response doesn't match the schema, the system enters a repair chain: `normalizeOptionalPacketFields()` strips empty strings and arrays, `isRecoverableModulePacketMismatch()` checks if the failure is a full-packet missing its required sections, `buildModuleRepairPrompt()` constructs a correction prompt with the validation errors, the model gets one more shot, and `coerceSummaryOnlyModuleGenerationResult()` attempts to salvage whatever it can by downgrading to summary-only mode.

The model cannot read source code during generation. It works from summaries of exports and relationships. It cannot decide that a module needs more or fewer sections than the scoring algorithm predicted. It cannot iterate on a diagram that doesn't make sense. Every module gets exactly one shot (plus one repair attempt), and the scoring algorithm makes the section-inclusion decision before the model even sees the prompt.

After this design ships, each module is generated by an agent with three tools: `read_source` to inspect actual code, `write_section` to build the page incrementally, and `report_observation` to flag taxonomy gaps. The agent decides what sections to include based on the code it reads, not a scoring algorithm's prediction. The agent can iterate: write an overview, read more code, revise the overview, add a diagram. The scoring algorithm, the repair chain, and the coercion logic are no longer in the generation path.

The agent runtime is a new module at `src/agent/runtime.ts`. It is not embedded in the generation stage. The generation stage calls into the runtime, passing module context and a configured provider. The runtime manages the tool-use conversation loop, collects section writes into a buffer, collects observations into a list, enforces timeout and turn limits, and returns structured output that the generation stage renders into markdown. This separation means the runtime can be tested independently of the generation pipeline, and the generation stage remains a thin orchestrator that iterates modules and handles per-module outcomes.

## High Altitude: Where the Agent Runtime Sits

The agent runtime occupies a specific position in the system. It sits between two well-defined interfaces: the provider's `inferWithTools()` on the downstream side (how it talks to the model) and the generation stage's module iteration loop on the upstream side (how it gets invoked). It does not know about the pipeline, the plan, or the analysis. It receives a fully assembled context and a set of tools, runs a conversation, and returns a result.

```
Generation Stage (module-generation.ts)
    │
    │  assembles module context from:
    │  - buildModuleDocumentationFacts() (existing, reused)
    │  - classification map (from td-strategy-classification)
    │  - module archetype
    │  - strategy zone guidance
    │
    ▼
Agent Runtime (src/agent/runtime.ts)
    │
    │  manages:
    │  - tool-use conversation loop via provider.inferWithTools()
    │  - section buffer (Map<PageSectionKind, string>)
    │  - observation collector (in-memory list)
    │  - timeout enforcement
    │  - turn counting
    │
    │  dispatches tool calls to:
    │  ├── read-source.ts    → reads file from repo, sandboxed
    │  ├── write-section.ts  → writes to section buffer
    │  └── report-observation.ts → appends to observation list
    │
    ▼
Agent Result
    │
    │  contains:
    │  - section buffer contents (Map<PageSectionKind, string>)
    │  - observations reported
    │  - turn count, tool call count
    │  - success/failure status
    │
    ▼
Generation Stage (continued)
    │
    │  assembles page via assembleAgentPage() (title + sections in order)
    │  validates output
    │  writes to disk
    │  records per-module outcome
```

The runtime does not import from `src/orchestration/`. The generation stage does not import from `src/agent/tools/`. The boundary is the `AgentModuleResult` type (defined below in Low Altitude), which carries the section buffer contents and observation list back to the generation stage.

This separation is load-bearing. The generation stage handles iteration, file I/O, progress reporting, and error recovery (placeholder pages, per-module outcomes). The runtime handles conversation management, tool dispatch, and safety limits. Neither reaches into the other's domain.

### Responsibility Map

### What Gets Reused

One existing function from `src/orchestration/module-doc-packet.ts` is reused in the agent path.

`buildModuleDocumentationFacts()` assembles the deterministic context the agent receives as its initial prompt. This function already extracts entity candidates from component exports and relationship edges, flow candidates from cross-module relationships, internal and cross-module relationship summaries, and source coverage paths. The agent gets all of this as structured context in its system prompt, exactly as the one-shot path does today. The function is not modified; it is called from the generation stage before invoking the runtime.

`renderModuleDocumentationPacket()` is NOT used in the agent path. That renderer expects structured data (typed arrays for entity tables and flow notes) which the agent does not produce. The agentic path uses a simple assembly function that concatenates pre-formatted markdown sections in canonical order with headings. See Flow A Step 5 below.

`selectModuleDocumentationPacket()` is NOT used in the agent path. The agent decides what sections to include. The scoring heuristics are bypassed entirely.

## Medium Altitude: Flow-by-Flow Design

### Flow A: Agent Generates a Module Page (AC-4.1 through AC-4.3)

This is the primary flow. The generation stage selects a module, assembles context, invokes the agent runtime, and renders the result.

**Step 1: Context Assembly (generation stage)**

The generation stage calls `buildModuleDocumentationFacts()` to get the same deterministic context the one-shot path uses. It then augments this with classification data from the classification map (component roles and zones) and the module's archetype from the module classifier. It also pulls zone guidance from the documentation strategy for this module's zone. All of this is assembled into an `AgentModuleContext` that the runtime consumes.

The context includes:
- Module name, description, and component list
- Entity candidates (name, kind, file path, public entrypoints, dependency edges)
- Flow candidates (actor, action, output, target, weight)
- Internal and cross-module relationship summaries
- Source coverage paths (all file paths in the module)
- Per-component role and zone labels
- Module archetype label
- Zone treatment guidance from the documentation strategy

This context becomes the agent's system prompt. The agent does not need to infer what the module contains from file paths alone — it has a structured vocabulary for what each component is and how they relate. But the agent can override the classification's judgment by reading the actual code and deciding that, say, a "utility" component is actually doing orchestration work.

**Step 2: Runtime Initialization**

The generation stage calls `runAgentForModule()` (exported from `runtime.ts`) with:
- The assembled `AgentModuleContext`
- The `InferenceProvider` (which must support tool use — the generation stage checks `provider.supportsToolUse()` before entering this path)
- An `AgentRuntimeConfig` (timeout, max turns, repo root path)
- The run's `ObservationCollector` instance (shared across all modules)

The runtime creates a fresh `SectionBuffer` for this module. It builds the tool definitions array (three tools with their JSON schemas). It constructs the system prompt by serializing the module context into a structured format that tells the agent what module it's documenting, what components are in it, what their roles are, and what tools are available. The system prompt also includes guidance on documentation quality: when to include diagrams, how to write entity tables, when sequence diagrams add value vs noise.

**Step 3: Tool-Use Conversation**

The runtime constructs `SdkMcpToolDefinition[]` via `buildSdkToolDefinitions()`, embedding handler functions that dispatch to the three tool executors. It then calls `provider.inferWithTools()` with these tool definitions, the system prompt, an initial user message, and the max turns cap (default 15). The provider creates an in-process MCP server from the tool definitions, starts a `query()` with built-in tools disabled, and returns a `ToolUseHandle` with a result promise and a cancel function.

The SDK manages the conversation loop internally. When the model requests a tool call, the SDK invokes the handler function embedded in the corresponding `SdkMcpToolDefinition`. Each handler wraps one of our tool executors:

- `read_source` handler: Calls `executeReadSource()` in `read-source.ts`. Resolves the file path relative to the repo root, validates it stays within the sandbox, reads the file, and returns content capped at 2000 lines. If the file exceeds the cap, the response includes a `truncated: true` flag and a notice showing how many lines were omitted.

- `write_section` handler: Calls `executeWriteSection()` in `write-section.ts`. Validates the `section` parameter against the `PageSectionKind` enum, then calls `sectionBuffer.set(section, content)`. Last-write-wins: if the agent writes the overview twice, the second write replaces the first. This is intentional — the agent may read more code and decide to revise a section.

- `report_observation` handler: Calls `executeReportObservation()` in `report-observation.ts`. Constructs an `AgentObservation` from the parameters (adding the current module name from context), appends it to the shared `ObservationCollector`, and returns `{ recorded: true }`. If the append fails for any reason, the handler catches the error and returns `{ recorded: true }` anyway — observations are fire-and-forget per AC-3.3.

Each handler also increments a tool call counter via the `onToolCall` callback passed to `buildSdkToolDefinitions()`. This is how the runtime tracks `toolCallCount` — the provider only reports turn count.

The conversation continues until either the model produces a final result (no more tool calls), the turn limit is reached, or the runtime's timeout fires (calling `handle.cancel()`).

**Step 4: Result Assembly**

After the conversation ends, the runtime assembles an `AgentModuleResult`:
- `sections`: the section buffer's current state, exported via `toSectionRecord()`
- `observationCount`: how many observations this module's agent reported
- `turnCount` and `toolCallCount`: from the provider's `ToolUseConversationResult`
- `status`: `"success"` if the section buffer contains all required sections (`overview` and `source-coverage` per AC-4.3c), `"failed"` otherwise
- `failureReason`: populated if status is failed (e.g., "Agent completed without writing required sections: source-coverage")

The runtime does not validate the content of sections beyond checking that required sections exist. Content validation (Mermaid syntax, cross-link resolution, entity table completeness) happens in the existing validation suite after the page is rendered and written to disk.

**Step 5: Rendering (generation stage)**

The generation stage receives the `AgentModuleResult`. If the status is `"success"`, it assembles the final markdown page from the section buffer output. The agentic path does NOT call `renderModuleDocumentationPacket()` — that renderer expects structured data (typed arrays for entity tables and flow notes), while the agent produces pre-formatted markdown.

Instead, a simple `assembleAgentPage(moduleName: string, sections: Record<string, string>): string` function handles assembly. It prepends the title (`# ${moduleName}`) — the title is not a section the agent writes, it's derived from the module plan. Then it concatenates sections in canonical order with headings. This satisfies AC-4.3c (title + overview + source coverage are required) without requiring the agent to write its own title.

The canonical order aligns with the existing renderer's output so pages look consistent regardless of generation path:

| PageSectionKind | ModuleGenerationResult field |
|-----------------|------------------------------|
| `overview` | `overview` |
| `responsibilities` | `responsibilities` (parsed as newline-delimited list) |
| `structure-diagram` | `structureDiagram` + `structureDiagramKind` |
| `entity-table` | `entityTable` (parsed as structured data) |
| `sequence-diagram` | `sequenceDiagram` |
| `flow-notes` | `flowNotes` (parsed as structured data) |
| `source-coverage` | (appended to page content) |
| `cross-module-context` | `crossLinks` (extracted from content) |

All section content is pre-formatted markdown written by the agent. The assembly function doesn't parse or transform content — it concatenates sections with headings (e.g., `## Overview\n\n` + overview content). This is intentional: the agent has read the actual code and can produce better formatting than the deterministic fallback generators (`defaultEntityTable`, `defaultFlowNotes`) that the one-shot renderer uses.

The one validation the generation stage performs before rendering: if the section buffer contains `flow-notes` but not `sequence-diagram`, the flow notes are stripped (per AC-4.3e). Flow notes without a sequence diagram are meaningless.

**Step 6: Write and Record**

The rendered markdown is written to disk. The generation stage records a `ModuleGenerationOutcome` with `generationPath: "agentic"`, `turnCount`, `toolCallCount`, and `observationCount`. The module is complete.

### Flow B: Agent Reports Observations (AC-3.1 through AC-3.3)

Observations are a secondary output of the agent conversation. They do not drive generation behavior. They are collected across all modules in a run and persisted once at the end as a diagnostic artifact.

**During Generation (per-module)**

The agent encounters a component whose classification doesn't match what it sees in the code. For example, it reads a file labeled `role: "utility"` and finds it's actually implementing a repository pattern with database queries and transaction management. It calls `report_observation` with:

```
category: "classification-gap"
subject: "src/data/user-store.ts"
observation: "Component is labeled 'utility' but implements a data repository pattern with query methods and transaction handling"
suggestedCategory: "repository"
```

The `report-observation.ts` handler constructs an `AgentObservation`:

```typescript
{
  moduleName: context.moduleName,        // from the runtime's current module context
  category: "classification-gap",
  subjectKind: "component",              // inferred from subject being a file path
  subject: "src/data/user-store.ts",
  observation: "Component is labeled 'utility' but implements ...",
  suggestedCategory: "repository"
}
```

It appends this to the shared `ObservationCollector` and returns `{ recorded: true }`. The agent continues generating the module page. The observation has no effect on section content, page rendering, or module success/failure status.

The `subjectKind` field is inferred by the handler:
- If `subject` contains a `/` or `.`, it's a `"component"` (file path)
- If `subject` matches a known module name from context, it's a `"module"`
- If `subject` contains `->` or `→`, it's a `"relationship"`
- Default: `"component"`

This inference is best-effort. The observation is informational; getting `subjectKind` wrong doesn't break anything.

**After All Modules Complete (run level)**

The generation stage (or the orchestrator above it) calls `observationCollector.persist(outputPath)` after all modules have been generated. If the collector has zero observations, `persist()` is a no-op — no file is written (AC-3.2b). If there are observations, it writes `.doc-observations.json` to the output directory:

```json
{
  "runId": "gen-2026-03-19-a7c4",
  "timestamp": "2026-03-19T14:23:07.000Z",
  "observationCount": 5,
  "observations": [
    {
      "moduleName": "Data Access",
      "category": "classification-gap",
      "subjectKind": "component",
      "subject": "src/data/user-store.ts",
      "observation": "Component is labeled 'utility' but implements ...",
      "suggestedCategory": "repository"
    }
  ]
}
```

The `persist()` method catches its own I/O errors. If writing fails, it logs a warning and returns without throwing. Observation persistence failure does not affect the run result (AC-3.3). The run can report `status: "success"` even if observation persistence failed.

**Observation Categories and Their Triggers**

| Category | When the agent reports it | Example |
|----------|--------------------------|---------|
| `classification-gap` | Component role doesn't exist in the taxonomy, or the assigned role is clearly wrong based on source code | A file doing event sourcing labeled `utility` |
| `zone-ambiguity` | Component zone doesn't match the code's actual nature | A test helper in `src/` (production zone) that's clearly test infrastructure |
| `relationship-gap` | The relationship between components isn't captured by the analysis | Two components communicate through a shared event bus that the static analysis can't see |
| `archetype-mismatch` | The module archetype doesn't fit the module's actual composition | A module labeled `type-definitions` that contains substantial runtime logic |

The agent is not instructed to look for observations. It reports them when it naturally encounters mismatches during the process of reading code and writing documentation. The system prompt mentions the `report_observation` tool and its purpose, but does not ask the agent to audit classifications. This keeps observation overhead minimal — the agent generates documentation first and reports gaps incidentally.

### Flow C: Agent Timeout and Cleanup (AC-4.5)

The agent operates under a per-module time budget. The default is 120 seconds, configurable via `AgentRuntimeConfig.timeoutMs`. The timeout is the primary budget constraint; the max turns cap (15) is a safety net against runaway conversations that make many fast tool calls.

**Timeout Mechanism**

The runtime starts a timer when `runAgentForModule()` is called. The timer calls `handle.cancel()` on the `ToolUseHandle` returned by `provider.inferWithTools()`. The handle's `cancel()` function calls `query.close()` on the underlying SDK query — the SDK's native cancellation mechanism that forcefully terminates the subprocess and cleans up all resources (MCP transports, pending requests).

When the timeout fires:

1. `handle.cancel()` calls `query.close()`, which terminates the SDK subprocess.

2. The `handle.result` promise resolves with an error `EngineResult` (the SDK throws an `AbortError` or similar when closed, which the provider catches and wraps).

3. The runtime receives the error result, checks the `cancelled` flag in the error details, and assembles an `AgentModuleResult` with `status: "failed"` and `failureReason: "Agent exceeded time budget (120000ms)"`.

4. Any partially written sections in the buffer are discarded. The generation stage will produce a placeholder page for this module (handled in `td-degradation-cleanup.md`).

5. The `setTimeout` timer is cleared in the `finally` block. No orphan timers remain.

The runtime does not attempt to salvage partial output from a timed-out agent. The rationale: a timed-out agent may have written an overview but not yet written the sections it planned to. Publishing a half-built page with an overview that references diagrams and entity tables that don't exist would be worse than a clean placeholder.

**Turn Limit**

If the agent reaches 15 turns without completing, the SDK stops the conversation (the `maxTurns` parameter passed through to `query()`). The `handle.result` promise resolves with a `ToolUseConversationResult` reflecting whatever the model's final state was. The runtime checks the section buffer using `hasRequiredSections()` — the same check used on normal completion. If the buffer contains both `overview` and `source-coverage`, the result is `"success"`. If not, it's `"failed"` with `failureReason` listing the missing required sections.

This is different from timeout behavior because the agent may have written usable content before hitting the turn limit. A timeout suggests the agent was stuck or the inference was slow; a turn limit suggests the agent was doing a lot of work. If that work produced a page with all required sections, it should be kept.

**Cleanup Guarantees (AC-4.5b)**

The runtime uses `try/finally` around the `await handle.result` to ensure:
- The timeout timer is cleared (`clearTimeout`)
- The section buffer and observation collector are in a consistent state (the collector is shared and doesn't need cleanup; the buffer is local to this module and is garbage-collected)

No child processes are spawned by the runtime. The provider and SDK manage subprocess lifecycle. The `query.close()` call (via `handle.cancel()`) handles SDK subprocess cleanup. The runtime's cleanup responsibility is limited to the timer.

### Flow D: Module Generation Loop with Per-Module Error Handling (AC-4.4)

This flow describes how the generation stage iterates modules and handles per-module failures. The runtime is a callee here; the generation stage is the actor.

**Current Behavior**

Today, `generateModuleDocs()` iterates modules in a `for...of` loop. If any module fails (the `contentResult` from `generateModulePage()` has `ok: false`), the function immediately returns the error, aborting all remaining modules. This is the behavior AC-4.4 eliminates.

**New Behavior**

The generation stage changes its loop to catch per-module failures:

1. For each module, the generation stage calls either `runAgentForModule()` (if the provider supports tool use) or the existing one-shot path.

2. If the agent runtime returns `status: "failed"` (or the one-shot path returns an error), the generation stage:
   - Creates a placeholder page (module name, component list, "generation failed" notice, failure reason)
   - Writes the placeholder to disk
   - Records a `ModuleGenerationOutcome` with `status: "failed"`, `failureReason`, and `hasPlaceholderPage: true`
   - Continues to the next module

3. If the agent runtime returns `status: "success"`, the generation stage renders the page, writes it to disk, and records a success outcome.

4. After all modules are processed, the generation stage computes the `RunStatus` from the outcomes (success/partial-success/failure per the rules in the epic).

The generation stage wraps each module's generation in a try/catch so that even unexpected errors (not just `EngineResult` failures) are caught and converted to failed outcomes. An unhandled exception in one module's agent does not propagate to the loop.

This is a significant change to `generateModuleDocs()`. Today it returns `EngineResult<GeneratedModuleSet>`. After this change, it returns `EngineResult<DocumentationRunResult>` where the result contains per-module outcomes, success/failure counts, and the overall run status. The `GeneratedModuleSet` map is still populated (successful modules get their entries; failed modules are omitted from the map but present in the outcomes array).

The detailed design of placeholder pages, run-level result assembly, and partial-success semantics belongs to `td-degradation-cleanup.md`. This document covers the runtime and tool mechanics; the degradation document covers the error handling policy.

## Low Altitude: Interface Definitions

### Agent-Domain Types (`src/agent/types.ts`)

```typescript

/**
 * Configuration for a single module's agent run.
 * Passed from the generation stage to the runtime.
 */
export interface AgentRuntimeConfig {
  /** Absolute path to the repository root. Used for read_source sandboxing. */
  repoRoot: string;

  /** Per-module timeout in milliseconds. Default: 120_000. */
```

### Section Buffer (`src/agent/section-buffer.ts`)

```typescript
import {
  PAGE_SECTION_KINDS,
  REQUIRED_SECTIONS,
  type PageSectionKind,
} from "./types.js";

/**
 * Accumulates section content during an agent conversation.
 * Last-write-wins per section kind.
 *
 * The buffer is local to a single module's agent run.
 * After the conversation ends, the generation stage reads the buffer
 * via toSectionRecord() and passes it to the renderer.
 *
 * References: AC-4.2 (section accumulation), AC-4.3e (flow-notes constraint)
 */
export class SectionBuffer {
  private readonly sections: Map<PageSectionKind, string> = new Map();

  /** Write a section. Replaces any previous content for this kind. */
  set(kind: PageSectionKind, content: string): void {
    this.sections.set(kind, content);
  }

  /** Read a section's current content, if any. */
  get(kind: PageSectionKind): string | undefined {
    return this.sections.get(kind);
  }

  /** Check whether a section has been written. */
  has(kind: PageSectionKind): boolean {
    return this.sections.has(kind);
  }

  /** Number of sections written. */
  get size(): number {
    return this.sections.size;
  }

  /**
   * Check whether all required sections have been written.
   * Required: overview, source-coverage (per AC-4.3c).
   */
  hasRequiredSections(): boolean {
    return REQUIRED_SECTIONS.every((kind) => this.sections.has(kind));
  }

  /** Returns the names of required sections that have not been written. */
  getMissingRequired(): PageSectionKind[] {
    return REQUIRED_SECTIONS.filter((kind) => !this.sections.has(kind));
  }

  /**
   * Export sections as a plain record, in canonical order.
   * Sections not written by the agent are omitted.
   *
   * Applies the flow-notes constraint (AC-4.3e):
   * flow-notes are stripped if no sequence-diagram is present.
   */
  toSectionRecord(): Record<string, string> {
    const record: Record<string, string> = {};

    for (const kind of PAGE_SECTION_KINDS) {
      if (!this.sections.has(kind)) {
        continue;
      }

      // AC-4.3e: flow notes without a sequence diagram are stripped
      if (kind === "flow-notes" && !this.sections.has("sequence-diagram")) {
        continue;
      }

      const content = this.sections.get(kind);
      if (content !== undefined) {
        record[kind] = content;
      }
    }

    return record;
  }
}
```

### Observation Collector (`src/agent/observation-collector.ts`)

```typescript
import { writeFile } from "node:fs/promises";
import path from "node:path";

import type { ObservationCategory } from "./types.js";

/**
 * Structured observation reported by an agent during module generation.
 *
 * References: AC-3.1 (observation structure), AC-3.2c (required fields)
 */
export interface AgentObservation {
  moduleName: string;
  category: ObservationCategory;
  subjectKind: "component" | "module" | "relationship";
  subject: string;
  observation: string;
  suggestedCategory?: string;

/**
 * The persisted shape of the observations file.
 *
 * References: AC-3.2a (file structure), AC-3.2c (run metadata)
 */
export interface RunObservations {
  runId: string;
  timestamp: string;
  observationCount: number;
  observations: AgentObservation[];
}

/**
 * Collects observations across all module agents in a run.
 * Shared instance passed to each module's agent runtime.
 *
 * Thread-safe in the sense that module generation is sequential
 * (no concurrent module agents), so no locking is needed.
 *
 * References: AC-3.2 (persistence), AC-3.3 (fire-and-forget)
 */
export class ObservationCollector {
  private readonly observations: AgentObservation[] = [];
  private readonly runId: string;

  constructor(runId: string) {
    this.runId = runId;
  }

  /**
   * Append an observation. Never throws.
   * If the input is malformed, the observation is silently dropped.
   */
  add(observation: AgentObservation): void {
    try {
      this.observations.push(observation);
    } catch {
      // Fire-and-forget per AC-3.3
    }
  }

  /** All observations collected so far. */
  getAll(): readonly AgentObservation[] {
    return this.observations;
  }

  /** Number of observations collected. */
  count(): number {
    return this.observations.length;
  }

  /**
   * Write observations to .doc-observations.json in the output directory.
   * No-op if no observations were collected (AC-3.2b).
   * Catches its own I/O errors — persistence failure does not propagate.
   *
   * References: AC-3.2a, AC-3.2b, AC-3.3
   */
  async persist(outputDir: string): Promise<void> {
    if (this.observations.length === 0) {
      return;
    }

    const payload: RunObservations = {
      runId: this.runId,
      timestamp: new Date().toISOString(),
      observationCount: this.observations.length,
      observations: this.observations,
    };

    try {
      const filePath = path.join(outputDir, ".doc-observations.json");
      await writeFile(filePath, JSON.stringify(payload, null, 2), "utf8");
    } catch {
      // Observation persistence failure is non-fatal per AC-3.3
    }
  }
}
```

### Read Source Tool (`src/agent/tools/read-source.ts`)

```typescript
import { readFile } from "node:fs/promises";
import path from "node:path";

import type {
  AgentRuntimeConfig,
  ReadSourceInput,
  ReadSourceOutput,
} from "../types.js";

/**
 * Execute a read_source tool call. Returns file content sandboxed
 * to the repository root.
 *
 * Sandbox enforcement (AC-4.1b):
 * - Resolves the path relative to repoRoot
 * - Normalizes with path.resolve to collapse ../
 * - Rejects if the resolved path doesn't start with repoRoot
 *
 * Line cap (from index Q3 answer):
 * - Returns first maxReadLines lines (default 2000)
 * - If file exceeds cap, returns truncation notice with omitted line count
 *
 * References: AC-4.1a (read component source), AC-4.1b (sandbox),
 *             AC-4.1c (multiple reads)
 */
export function executeReadSource(
  input: ReadSourceInput,
  config: AgentRuntimeConfig,
): Promise<ReadSourceOutput> {
  return readSourceFile(input.filePath, config);
}

async function readSourceFile(
  filePath: string,
  config: AgentRuntimeConfig,
): Promise<ReadSourceOutput> {
  const resolvedPath = path.resolve(config.repoRoot, filePath);

  // Sandbox check: resolved path must be within repo root
  if (!resolvedPath.startsWith(config.repoRoot + path.sep) &&
      resolvedPath !== config.repoRoot) {
    return {
      error: `Path "${filePath}" resolves outside the repository root. Access denied.`,
    };
  }

  try {
    const content = await readFile(resolvedPath, "utf8");
    const lines = content.split("\n");

    if (lines.length <= config.maxReadLines) {
      return {
        content,
        lineCount: lines.length,
        truncated: false,
      };
    }

    const truncatedContent = lines.slice(0, config.maxReadLines).join("\n");
    const omittedCount = lines.length - config.maxReadLines;

    return {
      content: truncatedContent +
        `\n\n--- Truncated: ${omittedCount} lines omitted (file has ${lines.length} total lines) ---`,
      lineCount: config.maxReadLines,
      truncated: true,
    };
  } catch (error) {
    const message = error instanceof Error ? error.message : String(error);
    return {
      error: `Failed to read "${filePath}": ${message}`,
    };
  }
}
```

### Write Section Tool (`src/agent/tools/write-section.ts`)

```typescript
import { PAGE_SECTION_KINDS, type PageSectionKind } from "../types.js";
import type { SectionBuffer } from "../section-buffer.js";
import type { WriteSectionInput, WriteSectionOutput } from "../types.js";

/**
 * Execute a write_section tool call. Validates the section kind and
 * writes to the buffer. Last-write-wins.
 *
 * References: AC-4.2 (agent decides sections), AC-4.3 (valid output)
 */
export function executeWriteSection(
  input: WriteSectionInput,
  buffer: SectionBuffer,
): WriteSectionOutput | { error: string } {
  if (!isValidSectionKind(input.section)) {
    return {
      error: `Invalid section kind "${input.section}". Valid kinds: ${PAGE_SECTION_KINDS.join(", ")}`,
    };
  }

  buffer.set(input.section, input.content);
  return { written: true };
}

  return (PAGE_SECTION_KINDS as readonly string[]).includes(value);
}
```

### Report Observation Tool (`src/agent/tools/report-observation.ts`)

```typescript
import type { ObservationCollector, AgentObservation } from "../observation-collector.js";
import type { ReportObservationInput, ReportObservationOutput } from "../types.js";

/**
 * Execute a report_observation tool call. Appends to the shared collector.
 * Never throws — observations are fire-and-forget.
 *
 * References: AC-3.1 (agent reports observations), AC-3.3 (no failure propagation)
 */
export function executeReportObservation(
  input: ReportObservationInput,
  moduleName: string,
  collector: ObservationCollector,
): ReportObservationOutput {
  try {
    const observation: AgentObservation = {
      moduleName,
      category: input.category,
      subjectKind: inferSubjectKind(input.subject),
      subject: input.subject,
      observation: input.observation,
      suggestedCategory: input.suggestedCategory,
    };

    collector.add(observation);
  } catch {
    // Fire-and-forget per AC-3.3b
  }

  return { recorded: true };
}

/**
 * Best-effort inference of subject kind from the subject string.
 * Not critical — observations are informational.
 */
function inferSubjectKind(
  subject: string,
): "component" | "module" | "relationship" {
  if (subject.includes("->") || subject.includes("\u2192")) {
    return "relationship";
  }

  if (subject.includes("/") || subject.includes(".")) {
    return "component";
  }

  return "module";
}
```

### Runtime Entry Point (`src/agent/runtime.ts`)

```typescript

import { tool, createSdkMcpServer, type SdkMcpToolDefinition } from "@anthropic-ai/claude-agent-sdk";
import { z } from "zod/v4";
import { ObservationCollector } from "./observation-collector.js";
import { SectionBuffer } from "./section-buffer.js";
import { executeReadSource } from "./tools/read-source.js";
import { executeReportObservation } from "./tools/report-observation.js";
import { executeWriteSection } from "./tools/write-section.js";
import type {
  AgentModuleContext,
  AgentModuleResult,
  AgentRuntimeConfig,
  AgentToolCall,
} from "./types.js";

/**
 * Run an agent to generate a single module's documentation page.
 *
 * This is the primary entry point for the agent runtime.
 * Called by the generation stage for each module when the provider
 * supports tool use.
 *
 * The function:
 * 1. Creates a fresh SectionBuffer for this module
 * 2. Builds tool definitions and the system prompt
 * 3. Calls provider.inferWithTools() with a tool executor callback
 * 4. Returns an AgentModuleResult with sections, metrics, and status
 *
 * References: AC-4.1 through AC-4.3 (generation), AC-4.5 (timeout),
 *             AC-3.1 through AC-3.3 (observations)
 */
export async function runAgentForModule(
  context: AgentModuleContext,
  provider: InferenceProvider,
  config: AgentRuntimeConfig,
  collector: ObservationCollector,
): Promise<AgentModuleResult> {
  const buffer = new SectionBuffer();
  let toolCallCount = 0;

  // Build SDK-native tool definitions with handlers that dispatch to our tool implementations.
  // Each handler wraps the tool executor and counts calls.
  const tools = buildSdkToolDefinitions(config, buffer, context.moduleName, collector, () => {
    toolCallCount++;
  });

  const systemPrompt = buildAgentSystemPrompt(context);

  // Start the tool-use conversation. Returns a handle with result promise + cancel function.
  const handle = provider.inferWithTools({
    systemPrompt,
    userMessage: `Generate the documentation page for the "${context.moduleName}" module.`,
    tools,
    maxTurns: config.maxTurns,
  });

  // Timeout enforcement: cancel the query if the time budget is exceeded.
  // Uses query.close() via the handle's cancel() function (see td-provider-tool-use.md).
  const timer = setTimeout(() => handle.cancel(), config.timeoutMs);

  try {
    const conversationResult = await handle.result;

    if (!conversationResult.ok) {
      const isCancelled = conversationResult.error.details?.cancelled === true;
      return {
        status: "failed",
        failureReason: isCancelled
          ? `Agent exceeded time budget (${config.timeoutMs}ms)`
          : `Agent conversation failed: ${conversationResult.error.message}`,
        sections: {},
        observationCount: countObservationsForModule(collector, context.moduleName),
        turnCount: 0,
        toolCallCount,
        usage: { inputTokens: 0, outputTokens: 0 },
        costUsd: null,
      };
    }

    const result = conversationResult.value;
    const hasRequiredSections = buffer.hasRequiredSections();
    const missingSections = buffer.getMissingRequired();

    return {
      status: hasRequiredSections ? "success" : "failed",
      failureReason: hasRequiredSections
        ? undefined
        : `Agent completed without writing required sections: ${missingSections.join(", ")}`,
      sections: buffer.toSectionRecord(),
      observationCount: countObservationsForModule(collector, context.moduleName),
      turnCount: result.turnCount,
      toolCallCount,
      usage: result.usage ?? { inputTokens: 0, outputTokens: 0 },
      costUsd: result.costUsd,
    };
  } catch (error) {
    const message = error instanceof Error ? error.message : String(error);
    return {
      status: "failed",
      failureReason: `Agent runtime error: ${message}`,
      sections: {},
      observationCount: countObservationsForModule(collector, context.moduleName),
      turnCount: 0,
      toolCallCount,
      usage: { inputTokens: 0, outputTokens: 0 },
      costUsd: null,
    };
  } finally {
    clearTimeout(timer);
  }
}
```

### Implementation Notes for Chunk 4

The runtime tests (C4.12) need a mock provider that implements `inferWithTools`. The mock simulates a multi-turn conversation by executing a scripted sequence of tool calls. This mock is distinct from the existing mock provider used in one-shot tests. It should be defined in `test/agent/fixtures.ts` alongside the mock module context.

A representative mock script for a happy-path test:

```typescript
// Mock provider script: simulate agent reading a file, then writing sections
const mockConversation: MockToolCallSequence = [
  { name: "read_source", input: { filePath: "src/example.ts" } },
  { name: "write_section", input: { section: "overview", content: "# Overview\n..." } },
  { name: "write_section", input: { section: "entity-table", content: "| Name | Kind | Role |\n..." } },
];
```

The mock provider simulates the SDK's behavior: it invokes the handlers embedded in each `SdkMcpToolDefinition` for the scripted tool calls, then resolves the `handle.result` promise with a `ToolUseConversationResult` containing the accumulated metrics.

The generation stage changes (C4.15) touch `generateModuleDocs()` but not the existing one-shot `generateModulePage()` path. The one-shot path remains as-is until Story 6 (scoring removal). The routing logic checks `provider.supportsToolUse()` at the top of `generateModuleDocs()` and branches accordingly:

```typescript
// Routing decision in generateModuleDocs
const useAgenticPath = provider.supportsToolUse();

for (const module of modules) {
  const result = useAgenticPath
    ? await generateModuleAgentic(module, ...)
    : await generateModulePage(module, ...);  // existing one-shot path
  // ... handle result, record outcome
}
```

This routing is per-run, not per-module. All modules in a run use the same path. The provider either supports tool use or it doesn't.

## Design Decisions and Rationale

### Why a Separate Runtime Module?

The agent conversation loop could live inside `generateModulePage()`. It wasn't placed there for three reasons.

First, testability. The runtime can be tested with a mock provider and no filesystem (beyond temp dirs for read-source tests). The generation stage tests can mock the runtime's return value. Neither test suite needs to set up the other's infrastructure.

Second, separation of concerns. The generation stage handles iteration, file I/O, progress reporting, and the `EngineResult` error contract. The runtime handles conversation management, tool dispatch, and safety limits. Merging them would create a 300+ line function that does both.

Third, future extensibility. If sub-agent decomposition is added later (out of scope per the epic), the runtime's interface is already clean enough to call recursively or to spawn parallel agent conversations for sub-modules.

### Why Last-Write-Wins for Sections?

The alternative is append mode, where each `write_section` call adds to the section's content. Last-write-wins was chosen because:

- The agent may read more code after writing an initial overview and decide to revise it. Append would require a separate "clear section" tool.
- The agent may write a structure diagram, realize it's wrong after reading more code, and want to rewrite it. With append, it would need to clear and rewrite.
- Last-write-wins is simpler to implement and simpler for the agent to reason about. "When you call write_section, you're setting the section's content" is clearer than "you're appending to it."

The risk is that the agent accidentally overwrites good content with worse content. This is accepted because the agent can also choose not to overwrite — it only calls `write_section` when it wants to. And the system prompt can guide the agent to read sections before overwriting them if it needs to preserve earlier content.

### Why 2000 Lines, Not Line Ranges?

The index document (Q3) decided on full file content with a 2000-line cap, not line-range reads. The rationale:

- Most source files in the target codebases are under 500 lines. The 2000-line cap covers the vast majority of files without any complexity.
- Line-range reads require the agent to know what line range is interesting before reading, which requires either a table-of-contents tool or a two-pass approach (read first N lines to see the structure, then read the interesting range). This adds tool calls and conversation turns for a case (files over 2000 lines) that is rare.
- The 2000-line cap still returns useful content for large files. The agent gets the first 2000 lines (typically the imports, type definitions, and first several functions) with a truncation notice telling it how much was omitted.

If large file support becomes important, a future enhancement can add an optional `lineRange` parameter to the `read_source` tool without breaking the existing interface.

### Why Fire-and-Forget for Observations?

Observations are diagnostic output, not control flow. Making them fire-and-forget (AC-3.3) means:

- The agent never blocks on observation persistence
- An observation tool error doesn't fail the module
- The collector's `add()` is wrapped in try/catch, so even if the collector itself is in a bad state, the agent continues
- The `persist()` call after all modules is also wrapped in try/catch

The trade-off is that observations can be silently lost. This is acceptable because observations are a feedback mechanism for humans reviewing classification quality, not a critical pipeline output. Losing one observation is not worse than never having had the observation at all.

### Why Not Validate Section Content in the Runtime?

The runtime checks that required sections exist (`overview` and `source-coverage`) but does not validate their content. Content validation happens after the page is rendered and written to disk, in the validation suite.

This separation is intentional. The runtime's job is conversation management and section collection. Content validation is a separate concern that applies equally to agentic and one-shot pages. The validation suite already handles Mermaid syntax and cross-link resolution. However, as noted in the "New Validation Checks" section below, the existing suite does NOT cover all the epic's validation requirements — new checks are needed.

### New Validation Checks Required

The existing validation suite (`src/validation/validate.ts`) runs five checks: file presence, cross-links, metadata shape, module tree, and Mermaid syntax. Several AC-4.3 requirements are not covered by these existing checks and need new validation check modules:

| AC | What Needs Checking | Existing Coverage | New Check Needed |
|----|--------------------|--------------------|------------------|
| AC-4.3c | Required sections (title, overview, source-coverage) present | Title is prepended by assembly; overview and source-coverage are enforced by runtime `REQUIRED_SECTIONS`. Validation should verify the assembled page has these. | `checkRequiredSections` — parse the page markdown and verify `# Title`, `## Overview`, and `## Source Coverage` headings exist |
| AC-4.3d | Entity table rows have non-empty name, kind, role | Not covered | `checkEntityTableRows` — if an entity table section exists, parse the markdown table and verify each row has non-empty values in the name, kind, and role columns |
| AC-4.3e | Flow notes accompanied by sequence diagram | Handled by `SectionBuffer.toSectionRecord()` which strips flow-notes without sequence-diagram. Validation should verify the assembled page. | `checkFlowNotesConstraint` — if flow notes section exists in the page, verify a sequence diagram section also exists |
| AC-4.3f | Source coverage lists actual component paths | Not covered | **Assembly-time check** in `assembleAgentPage()`, not a validation check. The generation stage has the component list in `AgentModuleContext` when it assembles the page. It parses the source-coverage section, extracts listed paths, and verifies each exists in `context.componentPaths`. Invalid paths are stripped with a warning added to the run result. This avoids a pipeline-order problem: the validation suite runs before `.module-plan.json` is written to disk, so a disk-based check couldn't access the component list. |

Three new validation checks are added to `src/validation/checks/` and wired into the `validateDocumentation` function alongside the existing five. They run in the same `Promise.all()` parallel execution. (AC-4.3f is handled at assembly time, not in the validation suite — see the table above.)

**New files:**

```
src/validation/checks/
├── required-sections.ts      # AC-4.3c
├── entity-table-rows.ts      # AC-4.3d
└── flow-notes-constraint.ts  # AC-4.3e
```

AC-4.3f (`checkSourceCoverage`) is NOT a validation check — it runs at assembly time in the generation stage. See the table above.

Each check follows the same pattern as existing checks: receives the output directory path, reads relevant files, returns `ValidationFinding[]`. Severity depends on whether the finding represents a contract violation or a quality concern:

| Check | Severity | Rationale |
|-------|----------|-----------|
| `checkRequiredSections` — missing overview or source-coverage | `error` | AC-4.3c defines these as required; absence is a contract violation |
| `checkEntityTableRows` — row with empty name/kind/role | `warning` | AC-4.3d says entries should have non-empty fields, but a weak row doesn't invalidate the page |
| `checkFlowNotesConstraint` — flow notes without sequence diagram | `warning` | The section buffer already strips these at assembly; this catches edge cases where assembly was bypassed |
| `checkSourceCoverage` — listed path not in component list | `warning` | May indicate stale content rather than a structural failure |

Missing required sections produce `error` findings, which cause the validation result status to be `fail`. This means a page without an overview will fail validation — consistent with the epic's requirement that these sections are mandatory. Quality checks produce `warning` findings, which result in `warn` status but don't block the run.

These checks ship in Chunk 4 alongside the agent runtime, since they validate output from both the agentic and one-shot paths. Adding them to the validation suite benefits the one-shot path too — it currently has no section-level validation.

**Validation contract update:** The existing `validationFindingSchema` in `src/contracts/validation.ts` defines `category` as a Zod enum: `"missing-file" | "broken-link" | "metadata" | "module-tree" | "mermaid"`. The three new validation checks require three new categories. The `ValidationFinding` type in `src/types/validation.ts` must be updated in parallel.

Both changes are Chunk 0 deliverables — the schema and type expansions ship in the foundation pass so that the new check modules can reference the categories from day one.

```typescript
// src/contracts/validation.ts — updated category enum
category: z.enum([
  "missing-file",
  "broken-link",
  "metadata",
  "module-tree",
  "mermaid",
  // New categories for section-level validation (Chunk 0)
  "required-section",    // AC-4.3c: missing overview or source-coverage
  "entity-table",        // AC-4.3d: malformed entity table rows
  "flow-notes",          // AC-4.3e: flow notes without sequence diagram
]),
```

```typescript
// src/types/validation.ts — updated category union
export interface ValidationFinding {
  severity: "error" | "warning";
  category:
    | "missing-file"
    | "broken-link"
    | "metadata"
    | "module-tree"
    | "mermaid"
    // New (Chunk 0)
    | "required-section"
    | "entity-table"
    | "flow-notes";
  message: string;
  filePath?: string;
  target?: string;
}
```

Note: `source-coverage` is NOT a validation category. AC-4.3f is handled at assembly time (see table above), not by the validation suite.
